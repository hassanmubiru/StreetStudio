# @streetjs/auth — authentication, sessions, tokens, API keys

- **Package:** `@streetjs/auth`
- **Consumers (StreetStudio):** member auth, session management, API keys, SDK bearer/apiKey
- **Depends on:** `@streetjs/core`, `@streetjs/database`, `@streetjs/security`
- **Wave:** 3 (web & identity)

## Motivation

StreetStudio needs real authentication — Argon2id hashing, sessions, refresh
tokens, hashed API keys, invalidation. This is generic platform security, not
product logic.

## Required API surface

- Password hashing: Argon2id `hash`/`verify` with tuned parameters.
- Sessions: issue/verify/invalidate; secure-cookie and/or JWT strategies.
- Refresh tokens: persistent, rotating, revocable.
- API keys: create (one-time secret reveal), securely hashed at rest, verify, revoke.
- Auth middleware for `@streetjs/http` producing an authenticated principal.

## Acceptance criteria

- [ ] Passwords are hashed with Argon2id; verification is constant-time; plaintext is never stored.
- [ ] Sessions can be issued, verified, and invalidated; invalidated sessions/tokens are rejected.
- [ ] Refresh tokens rotate on use and can be revoked; reuse of a rotated token is detected.
- [ ] API key secret is shown once at creation and only a hash is persisted; verify matches; revoke stops access.
- [ ] Auth failures are non-disclosing (no account enumeration).

## Non-goals

- No authorization decisions (that is `@streetjs/rbac`); no product membership model.
